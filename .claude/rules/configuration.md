# Configuration

- **`config.yaml`** — single source for app metadata, model names per provider, scanner exclude patterns, and the service URL map.
- **`.env`** — secrets, loaded by `ag_core/config.py` (walks up the tree to find it; the walk is disabled outright under `GENIUS_SECURE_DEFAULTS` — only an explicitly pinned `GENIUS_ENV_FILE` loads): `SKILL_API_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GROK_API_KEY`, `GIT_USERNAME`, `GIT_TOKEN`.
- **`GENIUS_DB_PATH`** — overrides the SQLite path (default `genius.db` at repo root; Docker maps it into the volume).
- **`GENIUS_DB_MAX_ROWS`** / **`GENIUS_DB_RETENTION_DAYS`** — history retention for the append-only `conversations`/`agent_logs` tables (`ag_core/utils/db.py`). `GENIUS_DB_MAX_ROWS` caps rows per table, trimming the OLDEST (default 100000, a generous ceiling that leaves a normal DB untouched; `0` disables). `GENIUS_DB_RETENTION_DAYS` additionally drops rows older than N days (default `0` = off). Pruning runs at `init_db()` startup (non-fatal) and via `db.prune_history()`; `seen_jtis` is excluded (already bounded by the JWT anti-replay DELETEs). `init_db()` also creates `idx_agent_logs_task_id` and `idx_agent_logs(agent_name, status)` for the per-task status UPDATEs and the dashboard busy check.
- **`GENIUS_JOBS_MAX_JOBS`** / **`GENIUS_JOBS_RETENTION_DAYS`** — retention for the MCP per-job workspaces under the jobs root (`GENIUS_JOBS_DIR`, default `.genius_jobs`): keep the newest N job dirs (default 50, `0` disables) and/or drop dirs older than N days (default `0` = off). Pruned on each `orchestrate` call; only 32-hex job-id dirs are candidates and unfinished in-memory jobs are never touched.
- **`GENIUS_ADAPTIVE_EFFORT`** (+ **`GENIUS_ADAPTIVE_EFFORT_THRESHOLD`**, **`GENIUS_ADAPTIVE_EFFORT_SMALL`**) — opt-in small-project effort downgrade in `orchestrator._resolve_pipeline_setup`: when truthy and the @modifier-stripped pipeline prompt is shorter than the threshold (default 600 chars), the pipeline-wide effort contextvar is set to the small level (default `high`, validated against low/medium/high/xhigh/max) — overriding per-role env efforts exactly like `@deep`; an explicit `@deep` wins, larger prompts are byte-identical. Off by default (and thus under pytest).
- **`GENIUS_PROGRESS_POLL_SECONDS`** — interval (default 5s) at which the MCP per-job watcher checks artifacts to PUSH `notifications/message` progress events over the live stdio session.
- **Bind addresses** — each server has a specific env var, with **`GENIUS_BIND_HOST`** as the generic fallback when the specific one is unset (blank = unset everywhere). All bare-metal services default to `127.0.0.1`. Docker/LAN deployments opt in to `0.0.0.0` through `GENIUS_SKILL_HOST`, `GENIUS_HUB_HOST`, `GENIUS_DASHBOARD_HOST`, or `GENIUS_MCP_HOST`; keep the corresponding authentication enabled whenever a service is exposed. `serve.py --distributed` prints a plaintext-transport warning on a non-loopback hub bind.
- **`GENIUS_CONFIG_PATH`** / **`GENIUS_ENV_FILE`** — explicit trusted config: pin exactly WHICH `config.yaml` / `.env` `ag_core.config` loads and disable the cwd-upward walk, so running Genius inside an untrusted repo cannot let that repo's files reconfigure provider paths, service URLs, or behavior toggles. An explicit `load_config(path)` argument still wins over `GENIUS_CONFIG_PATH`; a missing `GENIUS_ENV_FILE` loads nothing (never falls back to the walk). Unset = the legacy walk, unchanged (tests: `tests/test_config_trusted_paths.py`). Under `GENIUS_SECURE_DEFAULTS` both pins must be **absolute paths to existing files**, the `.env` cwd-walk is disabled entirely (no pin = load nothing), and `load_config` fail-closes with `RuntimeError` instead of walking for `config.yaml` or silently falling back to defaults on a missing pin.
- **`GENIUS_HUB_ADMIN_KEY`** / **`GENIUS_HUB_WORKSPACE_WRITE`** (+ **`GENIUS_HUB_WORKSPACE_ROOT`**) — hub credential separation (admin `X-Admin-Key` header for `/update_config`, `/tasks`, `/write_workspace_file`) and the write-endpoint gate (off outside pytest unless enabled; root pinned). Details in security.md.
- **`GENIUS_HUB_TLS`** — distributed workers dial `wss://` instead of `ws://` (for a hub behind a TLS-terminating proxy). **`GENIUS_WS_TOKEN_QUERY`** — legacy compat only: put the worker JWT back in the WS query string for hubs that predate Authorization-header auth.
- **`GENIUS_SECURE_DEFAULTS`** (+ **`GENIUS_ALLOW_PLAINTEXT`**) — opt-in production profile (`ag_core/security_profile.py`, default off). When truthy, **every entrypoint** FAILS CLOSED at startup via the shared `enforce_secure_defaults` gate (`serve.py`, `mcp_server.py` — both stdio and HTTP —, the `orchestrator.py` CLI, `dashboard.py`, `control_panel.py`; on stdio the refusal goes to stderr so stdout stays pure JSON-RPC) unless the fail-open conveniences are explicitly overridden: `GENIUS_CONFIG_PATH` must be an **absolute path to an existing file** (no cwd config walk, no silent fallback to defaults), `GENIUS_ENV_FILE` — when set — likewise, and starting the hub (`--distributed`) additionally requires `GENIUS_HUB_ADMIN_KEY` plus an explicit transport decision — `GENIUS_HUB_TLS=1` (a TLS proxy fronts the hub) or `GENIUS_ALLOW_PLAINTEXT=1` (knowingly accept plaintext). The transport gate is an explicit assertion, not a bind-address check, because a container legitimately binds `0.0.0.0` while its port is published only on loopback. Trust bootstrap: `GENIUS_SECURE_DEFAULTS` itself must come from the **real process environment** — the checker compares against the pre-dotenv snapshot `config._original_env`, so a workspace `.env` that flips the flag (and plants a pin satisfying the path checks) is refused; in secure mode `ag_core.config` also never walks for a `.env` at all. Off by default so local / trusted-LAN runs are byte-identical (tests: `tests/test_secure_defaults.py`). The `/write_workspace_file` gate (`GENIUS_HUB_WORKSPACE_WRITE`) is now env-only — it consults NO pytest signal (a stray `PYTEST_CURRENT_TEST` in the hub env can't open it; the suite enables it via `conftest.py`).
- **`GENIUS_TASK_PERSIST`** — opt-in durable skill-server task/idempotency state (default off = byte-identical in-memory behavior). When truthy, every task state transition is journaled to SQLite (`skill_tasks`/`skill_idempotency`, bounded per role at `MAX_TRACKED_TASKS`) and restored at lifespan startup: pollers hitting a restarted server get a real terminal status instead of a 404, `processing`-at-crash tasks come back as `failed: skill server restarted…`, and idempotent /run retries map back to their original task instead of re-running the agent (tests: `tests/test_task_persist.py`).
- **`GENIUS_CONTEXT_TOKEN_BUDGET`** — token budget for the workspace context each agent call receives (default 32000, `0` disables). `ag_core/scanner/repo_graph.py` ranks files with a personalized PageRank over the intra-repo import/reference graph (seeds: files the task names ×50, files defining mentioned identifiers ×10 — aider repo-map coefficients), then spends the budget in rank order: seeds full, others full while they fit, then `ast` signature skeletons. Workspaces under the budget pass through **unchanged** (identity), so small projects and the test suite see identical behavior; internal failures also fall back to the untrimmed dict.
- **tree-sitter (optional at runtime, pinned in requirements)** — `ag_core/scanner/code_parse.py` parses JS/TS/TSX/Go for the code graph (repo_graph edges, `graph_index.RepoIndex`, the MCP `code_graph` tool) through the OFFICIAL `tree-sitter` core + per-grammar wheels (`tree-sitter-javascript`/`-typescript`/`-go`). Probed with `find_spec`: when uninstalled, non-Python files simply contribute no graph info (never an error). Do NOT swap in `tree-sitter-language-pack`: its 1.x line replaced the classic py-tree-sitter node API with an undocumented PyO3 binding (`builtins.Tree`, `kind()`/`start_position()`), which breaks `code_parse`.
- **`GENIUS_CONFIG_CACHE_TTL`** — TTL (seconds) for the `load_config()` result cache (default 5, `0` disables). `load_config` sits on per-request hot paths (checksum middleware, JWT verification, response-checksum checks per status poll); the cache skips the repeated `.env`/`config.yaml`/registry re-reads. **Always disabled under pytest** (conftest re-points `SKILL_API_KEY` per test file). Within the TTL the returned `Config` object is shared — treat it as read-only.
- **Default provider chains** (no env vars needed; `DEFAULT_CHAINS` in `ag_core/provider_factory.py`): Researcher (role `researcher`) → `agy, claude, codex`; Architect (role `claude`) → `claude, agy, codex`; `codex`/`tester`/`security`/`devops` → `codex, claude, agy`. A backend that fails at runtime (`RuntimeError`/CLI timeout) is retried on the next backend, with sticky success within the process. The grok *backend* is in no default chain (opt-in only); the MCP `deploy` tool uses a claude-first default chain (`claude, codex, agy`). All construction sites (skill servers via `ag_core/skill_app.py`, `mcp_server.py`, distributed worker) build through `ag_core.agent_factory.build_agent`, which resolves the provider via `make_provider`; `python serve.py --doctor` prints each role's effective chain.
- **`GENIUS_PROVIDER_<ROLE>`** (e.g. `GENIUS_PROVIDER_RESEARCHER=grok,agy`) — explicit, comma-separated backend chain for one role (roles: `researcher`/`claude`/`codex`/`tester`/`security`/`devops`; backends: `grok`, `claude`, `codex`, `agy`, `notebooklm`). Overrides the default chain — including bringing the opt-in-only grok or notebooklm backends into play (e.g. `GENIUS_PROVIDER_RESEARCHER=notebooklm,agy,claude`). First backend is primary, the rest are runtime fallbacks; unknown names raise an actionable `ValueError`. Blank env values are treated as unset. The researcher role also honors the legacy `GENIUS_PROVIDER_GROK` spelling (from when the role id was `grok`); role ids `grok`/`grok_researcher` are compat aliases resolved by `canonical_role`.
- **`GENIUS_PROVIDER_FALLBACK`** — **deprecated no-op**: fallback chains are the default now, so this variable is accepted for backward compatibility but ignored (truthy or falsy makes no difference).
- **Model selection** — `config.models.<attr>` (attrs: `openai`, `anthropic`, `agy`; `config.models.grok` stays cosmetic; `config.models.notebooklm` is NOT a model name — it holds the default NotebookLM notebook id/alias, overridable by `GENIUS_MODEL_NOTEBOOKLM` or the friendlier `GENIUS_NOTEBOOKLM_NOTEBOOK`). Empty = the CLI's own default model; a value is passed through verbatim (`codex -m`, `claude --model`, `agy --model`). Env override per BACKEND name (not config attr): **`GENIUS_MODEL_<BACKEND>`** — `GENIUS_MODEL_CODEX` (maps to `models.openai`), `GENIUS_MODEL_CLAUDE`, `GENIUS_MODEL_AGY`; blank = unset. Never default `openai` to a chat-API name like `gpt-4o` — the codex CLI rejects it. The **grok** backend is the exception: it reads its own **`GENIUS_GROK_MODEL`** (empty by default → no `-m`, so the CLI uses its own default model; set e.g. `grok-composer-2.5-fast` to pin one — `grok models` lists valid, install-specific ids), not `GENIUS_MODEL_GROK`.
- **Per-ROLE model override** — `config.models.roles.<role>` (roles: `researcher`/`claude`/`codex`/`tester`/`security`/`devops`) and env **`GENIUS_MODEL_ROLE_<ROLE>`** (e.g. `GENIUS_MODEL_ROLE_RESEARCHER`) WIN over the per-backend model, resolved in `ag_core/provider_factory.build_backend`. Full precedence (blank = unset, `or`-chained): `GENIUS_MODEL_ROLE_<ROLE>` > `config.models.roles.<role>` > `GENIUS_MODEL_<BACKEND>` > `config.models.<attr>`. This lets two roles share ONE backend yet run different models — e.g. `GENIUS_PROVIDER_CODEX=agy` with `GENIUS_MODEL_ROLE_RESEARCHER=gemini-3.1-pro` + `GENIUS_MODEL_ROLE_CODEX=gemini-3.5-flash` gives the researcher and coder distinct Gemini models on the same agy backend. The env is deliberately `GENIUS_MODEL_ROLE_<ROLE>` (not `GENIUS_MODEL_<ROLE>`) so it never collides with the `GENIUS_MODEL_<BACKEND>` vars for the same-named claude/codex backends. With no per-role knob set, resolution is byte-identical to the per-backend path. The grok backend keeps its own `GENIUS_GROK_MODEL` (resolved in the grok provider, not `build_backend`), so a per-role override does NOT reach a role running on the grok backend.
- **`GENIUS_AGY_PATH`** — explicit path to the Antigravity `agy` CLI (backend `agy`, Gemini). Default resolution: PATH via `which_external`, then `%LOCALAPPDATA%\agy\bin\agy.exe`. No API key: auth is shared with the Antigravity IDE login. Model name comes from `config.models.agy` (empty = account default).
- **`GENIUS_AGY_SANDBOX`** — `agy` runs with `--sandbox` by default (print mode requires `--dangerously-skip-permissions`, so the sandbox is the remaining guardrail); set `0`/`false` to drop it. Its `--print-timeout` is derived from `GENIUS_CLI_TIMEOUT` minus 10s (floor 30s).
- **`GENIUS_NLM_PATH`** — explicit path to the NotebookLM `nlm` CLI (backend `notebooklm`, from notebooklm-mcp-cli — it lives in its own venv, so there is no cross-machine default: e.g. `...\notebooklm-mcp-cli\venv\Scripts\nlm.exe`). Default resolution: `GENIUS_NLM_PATH` > PATH via `which_external` > raises. No API key: auth is a one-time `nlm login` (or the `NOTEBOOKLM_COOKIES` env the CLI itself reads). The notebook the provider queries comes from `GENIUS_NOTEBOOKLM_NOTEBOOK` > `config.models.notebooklm`; with none set the provider raises (a `FallbackProvider` chain then falls through). Other knobs: **`GENIUS_NLM_QUERY_TIMEOUT`** (per-query seconds, default 120), **`GENIUS_NLM_RESEARCH_TIMEOUT`** (deep-research ceiling, default 900), **`GENIUS_NLM_MAX_QUERY_CHARS`** (truncates the question so it stays under the Windows argv limit, default 28000). Opt-in only (no default chain uses it); shared by the MCP `notebooklm_*` tools.
- **`GENIUS_CODEX_SANDBOX`** — Codex CLI sandbox policy: `read-only` (default — `--sandbox read-only --skip-git-repo-check`, Codex can reason but not execute/write), `workspace-write`, or `danger` (the old `--dangerously-bypass-approvals-and-sandbox`). Legacy `1`/`true`/`yes` and unknown values fail safe to read-only. `OpenAIProvider.send_prompt` also accepts a `workdir` kwarg that adds `--cd <dir>` to scope Codex's working root.
- **`GENIUS_CLAUDE_EFFORT`** / **`GENIUS_CLAUDE_FALLBACK_MODEL`** — opt-in Claude Code CLI passthroughs on the `claude` backend (`ag_core/providers/anthropic_provider.py`), both default off (argv unchanged unless set). `GENIUS_CLAUDE_EFFORT` maps to `--effort`; **Claude's reasoning-effort scale is `low`/`medium`/`high`/`xhigh`/`max` — there is NO "ultra" tier (that's a codex/`config.toml` concept); `max` is the ceiling.** An out-of-set value (e.g. a carried-over `ultra`) is ignored with a warning rather than passed through to fail the call. `GENIUS_CLAUDE_FALLBACK_MODEL` maps to `--fallback-model` (e.g. `claude-fable-5`) so an unavailable/declined primary (e.g. Opus 4.8) falls back to another model. Model itself comes from `GENIUS_MODEL_CLAUDE` / `config.models.anthropic` as usual. These apply to every claude-backend call (primarily the plan/architect stage, which is the main claude user).
- **`GENIUS_CODEX_EFFORT`** — opt-in reasoning-effort override for the `codex` backend (default off → uses `~/.codex/config.toml`). When set, `OpenAIProvider.send_prompt` appends `-c model_reasoning_effort=<value>` to `codex exec`, overriding the config.toml value (e.g. a global `ultra`) **for the pipeline's calls only** — Genius-scoped, so the user's global codex effort is untouched (same philosophy as `GENIUS_CODEX_LOGIN_MODE`). Passed through verbatim (codex validates it; common values incl. `low`/`medium`/`high`). Use to dial codex down from an expensive global default to a cheaper `high` for the code stage.
- **`GENIUS_CODEX_LOGIN_MODE`** — default **on**; keeps the codex backend on its native ChatGPT login by stripping `OPENAI_API_KEY` + `OPENAI_BASE_URL` from the **codex subprocess only** (`ag_core/providers/openai_provider.py::_codex_subprocess_env`, threaded through `cli_runner.spawn_cli(env=...)`). Codex is a login-based desktop CLI; if a machine also exports those vars (e.g. a LiteLLM / OpenAI-compatible proxy on `localhost:4000`), codex would silently route through the proxy in API-key mode instead of the ChatGPT-Plus login. Login mode hides them from codex without touching the process-wide proxy env (other tools still see it). A no-op when neither var is set (returns `None` → inherit). Set to `0`/`false`/`no`/`off` to inherit them (proxy / API-key mode).
- **`GENIUS_EVAL_GATE`** — opt-in R5 post-run eval gate (default off; **always off under pytest**). When truthy (`1`/`true`/`yes`), a finished `run_pipeline` grades its **workspace root** (where the artifacts live — grading the `projects/<slug>` deliverable left the artifact metrics blind) with the deterministic metric set (`ag_core/eval/gate.py` → `run_eval_gate`; offline — no judge/token spend), persists `score-<ts>.json`/`baseline.json` under `pipeline_internal_dir()/logs/eval/` (via `run_eval_gate(eval_dir=…)`; the default `<workspace>/logs/eval` remains for direct callers — never inside the deliverable), and compares against the baseline, logging a warning on a quality regression. The grader prunes `.genius`/`.genius_jobs` (generated tests + auto-install venv must not distort scores) and reads traces from `.genius/<slug>/logs/raw/` (legacy `logs/raw/` still honored). The baseline is preserved on a regressed run (so the regression stays visible until fixed). Fully non-fatal: a grading error is logged and never fails the run. The same engine is exposed read-only via the MCP `eval` tool (`op=grade`/`compare`/`list_metrics`).
- **`GENIUS_SURGICAL_EDITS`** — opt-in R5 code-preservation mode for the Codex generation path (default off). When truthy, `SURGICAL_EDIT_GUIDANCE` is appended to the coder system prompt for `/code`/`/refactor` requests: change only what the request requires, keep everything else byte-identical, and never touch model names / API keys / unrelated config. Off by default so the coder system prompt stays byte-identical to the pre-R5 behavior the tests pin.
- **`GENIUS_PROJECT_GATE`** — opt-in stack-aware project gates (default off; **always off under pytest**). When truthy, the custom flow's whole-project verification also detects the generated project's stack (v1: `package.json` → npm, Windows-aware `cmd.exe /c` wrap) and runs its OWN gates from the project root — `npm install --no-audit --no-fund` first, then whichever of the `test`/`lint`/`build` scripts the project declares — with `CI=1` (no watch mode / interactive wizards) and `GENIUS_INSTALL_TIMEOUT` per step. Results land in review.md under `## Project gates (stack-aware)`; any non-zero exit fails the job through the same strict full-suite gate (`GENIUS_FULL_SUITE_GATE=0` demotes to report-only; an `npm install` failure skips the dependent gates). SECURITY: npm install/scripts execute arbitrary project + registry code — enable only for runs you trust. Detection lives in `orchestrator.detect_project_gates` / `_run_project_gates` (tests: `tests/test_project_gate.py`).
- **`GENIUS_AUTO_INSTALL`** (+ **`GENIUS_INSTALL_TIMEOUT`**) — opt-in dependency auto-install (default off; **always off under pytest**). When truthy, root-level `requirements*.txt` files from the design are implemented in a wave 0 ahead of everything else (`orchestrator.partition_fanout_waves`, both flows), then `pip install --no-input -r`-ed into an isolated venv at `pipeline_internal_dir()/venv` (built `--system-site-packages` so pytest/flake8 stay importable; installed pins shadow inherited copies); every verification subprocess (per-file pytest, e2e flake8/pytest, the whole-project gate) then runs on that venv's interpreter via `verification_python(project_dir)` instead of `sys.executable`. Best-effort: venv/pip failures are logged (internal `logs/install.log`) and verification falls back to the orchestrator interpreter — missing deps then surface as ordinary test failures for the self-heal loop. `GENIUS_INSTALL_TIMEOUT` bounds each venv/pip subprocess (default 600s). SECURITY: installing LLM-generated requirements executes arbitrary package code at install time — enable only for runs you trust enough to install from. Off = byte-identical scheduling and interpreters (manifests are ordinary wave-1 files); a stale venv in a reused workspace is ignored while the flag is off.
- **`GENIUS_HACKATHON_MODE`** — opt-in hackathon mode for the CUSTOM flow (default off; **always off under pytest**). When truthy, rubric-oriented guidance constants (`HACKATHON_*` in `ag_core/utils/prompt_templates.py`) are appended to the research/design(plan-first + debate-refine)/devops *request* prompts — the architect's single-```json``` output contract is untouched (the guidance shapes DesignPlan field CONTENT only, and asks the design to ship a Dockerfile + one platform config so a live deploy URL is reachable). After the devops stage gate the pipeline additionally best-effort emits two workspace-root artifacts: `pitch.md` (one extra claude-role call over design/review/audit/deploy; raw-captured as `pitch`) and `ai_collaboration_log.md` (deterministic export from job.json + raw traces + review.md via `ag_core/collab_log.py`, also standalone: `python -m ag_core.collab_log <workspace> [--out PATH]`). Both are whitelisted MCP resources and stale-cleaned on reruns but are NOT stage checkpoints; a pitch/log failure logs and never fails the run (the log still exports when the pitch call died). The collab log's job metadata comes from `job.json`: MCP jobs journal it per state transition, and **CLI runs journal it too** (`orchestrator.run_cli_journaled` wraps both `python orchestrator.py` pipeline branches: `running` before, terminal `completed`/`failed` + `finished_at` after, best-effort/atomic via `_write_job_manifest`); after the final manifest write, both the CLI wrapper and the MCP completion path re-export an already-emitted log via `ag_core.collab_log.refresh_log_if_present` so the shipped log carries the terminal status instead of "running" (tests: `tests/test_cli_journal.py`). Off = byte-identical prompts, call sequence, and artifacts (tests: `tests/test_hackathon_mode.py`, `tests/test_collab_log.py`).
- `config.py` rewrites service URLs (adds `/role` suffixes) when running under pytest — be aware tests and production resolve URLs differently.
