# Architecture

## Request flow
```
Orchestrator  --async httpx + HMAC checksums-->  FastAPI Skill Server  -->  Agent  -->  Provider  -->  local CLI / API
```

- **`orchestrator.py`** — the async pipeline runner. `run_pipeline()` (sequential) and `run_e2e_pipeline()` chain the agents: Researcher (role id `researcher`, agy/Gemini-primary chain) → Claude (architect/design) → Codex (code) → Tester + Security + DevOps. Independent stages run concurrently via `asyncio.gather` (e.g. Tester and Security run in parallel per file in `process_single_file`). Handles tenacity retries, `Retry-After`-aware backoff, background-task polling, and response checksum verification. Per file, a self-healing loop (`--max-retries`, default 3) feeds Tester/Security failures back to Codex; an optional critic↔Claude design debate (`--max-debate-rounds`, default 2 but **0 under pytest**) exits early on an `[APPROVED]` marker. An unparseable design self-heals in production (`design_selfheal_enabled`, **off under pytest**): up to 2 format re-prompts, then `PipelineError` — the legacy single-file branch after it is a PYTEST-ONLY compatibility path, unreachable in production. Files that ARE pytest modules (`is_test_module`) are executed directly instead of getting tests-for-tests + a security audit. Raw agent responses are captured per stage/attempt under `projects/<slug>/logs/raw/` (`save_raw_response`). `run_pipeline(stage_gate=...)` awaits an async callback after the research/design/code stages (the MCP approval-gate hook).
- **`serve.py`** — unified entrypoint. Boots agent API servers by role, the central hub (`--distributed`), or delegates to the orchestrator. Owns the `WorkerRegistry` (idle-worker selection by role) and bounded pending-task tracking.
- **`ag_core/`** — the core library:
  - `interfaces/base_agent.py`, `interfaces/base_provider.py` — ABCs. `BaseProvider` carries a `TokenBucket` rate limiter + `asyncio.Semaphore(5)`; `BaseAgent` wires in `VectorMemory` and `GitManager`.
  - `directives.py` — the `@modifier` layer of the two-layer prompt syntax, separate from the `/cmd` routing/rewrite layers. `parse_directives(text)` strips a leading run of `@deep`/`@critic`/`@variants=N`/`@ideas=N`/format tokens (`@simple|table|steps|tight|natural|redpen`; legacy `/HUMAN`→`@natural`, `/FLOOD`→`@ideas`), returning a cleaned prompt whose leading token is still the `/cmd` (so both slash layers see byte-identical input) plus a frozen `PromptDirectives`. It is byte-preserving and returns the ORIGINAL string object when there is nothing to strip (no-op path — pinned by `test_slash_commands.py`). `@deep` sets `.effort="high"` (`DEEP_EFFORT`), threaded to the provider as a keyword-only `send_prompt(..., effort=...)` on the call stack — never via env — so concurrent jobs don't interfere; providers without an effort flag (agy/grok/notebooklm) accept-and-ignore it. `BaseAgent._resolve_user_prompt` parses + gates via the per-agent `ACCEPTED_MODIFIERS` knob (analogous to `SLASH_PREFIXES`): prose agents (researcher/devops) accept the full set; structure-sensitive stages (codex/tester/security/architect) accept only `{deep}` so `@table`/`@variants` can never perturb `extract_code`/`ast.parse`/`parse_security_verdict`/the DesignPlan JSON. Non-effort modifiers render as a trailing `_directive_guidance()` block. Wired on the DIRECT agent-call path (skill server / MCP single-agent / worker) AND pipeline-wide: `orchestrator._resolve_pipeline_setup` parses the leading @modifier run once, returning the cleaned prompt (used only for the slash-command routing DETECTION — `first_word`/`ROUTING_TABLE` in orchestrator + serve — so `@deep /code …` still routes to the single /code stage) and the `@deep` effort. The RAW prompt is still what agents receive (they re-parse @modifiers, so prose modifiers keep working), while the effort rides a per-task `_PIPELINE_EFFORT_VAR` contextvar (mirroring `_DISTRIBUTED_MODE_VAR`) that `call_api` reads and folds into the `/run` req_payload + distributed `task_data` (conditionally, before the HMAC, so a None effort is byte-identical); `skill_app.RunRequest.effort` and the worker forward it to `agent.run(effort=…)` — so `@deep` reaches codex/claude on structured (synthesized) stages even though those prompts carry no @deep token (agy research is a no-op, logged). Tests: `tests/test_directives.py`, `tests/test_directives_integration.py`, `tests/test_directives_pipeline.py`, per-request effort argv in `test_claude_effort.py`/`test_codex_effort.py`.
  - `agents/` — the six agent implementations (researcher, claude_architect, codex_reviewer, tester, security_agent, devops_agent; `grok_researcher.py` is a compat shim re-exporting `ResearcherAgent` as `GrokResearcherAgent`). Their shared request shape (resolve prompt → slash-command rewrite → scan → optional memory context → compose → send → history/memory/log/write) lives in `BaseAgent._run_standard` + class knobs (`SLASH_PREFIXES`, `SYSTEM_PROMPT`, `USES_MEMORY`, `DEFAULT_OUTPUT_FILE`); researcher/architect/security/devops just delegate, codex/tester keep custom `run()`s (verification / self-heal loops) on top of the same helpers. NOTE: `codex_reviewer` must keep a module-level `log_transaction` import — tests patch `ag_core.agents.codex_reviewer.log_transaction`.
  - `agent_factory.py` — the single agent-construction path (`build_agent(role, ...)`, stateless bundle `output_file="None"/use_memory=False/stateless=True`); `skill_app.ROLE_MAP` and the worker's `ROLE_AGENT_MAP` are derived from its `AGENT_CLASSES`/`LONG_ROLE_ALIASES` tables, and mcp_server passes `agent_cls=globals()[...]` so tests can keep patching `mcp_server.<AgentClass>`.
  - `scanner/` — `project_scanner.py` (`ProjectScanner` denylist walk; `ProjectChunker` token counting + greedy `chunk_files`), `repo_graph.py` (graph-aware token-budgeted context, `GENIUS_CONTEXT_TOKEN_BUDGET`), `code_parse.py` (per-language structural parsing: stdlib ast for Python, official tree-sitter grammars for JS/TS/TSX/Go when installed — fail-soft to empty info), `graph_index.py` (`RepoIndex`: definition/reference/import indexes + aider-style ranked `repo_map`, the engine behind the MCP `code_graph` tool).
  - `providers/` — LLM backends. Each `send_prompt` resolves a local CLI (Windows-aware: `.cmd`/`.bat` are wrapped with `cmd.exe /c`, falls back to `%APPDATA%\npm\`), runs it as a subprocess, and parses JSON/JSONL output. Every role defaults to a `FallbackProvider` chain from `ag_core/provider_factory.py` (Researcher: `agy → claude → codex`; Architect: `claude → agy → codex`; others: `codex → claude → agy`); the grok backend is opt-in only via `GENIUS_PROVIDER_<ROLE>` and auto-runs `grok login` (only when actually invoked) when no API key is present. `notebooklm_provider.py` (backend `notebooklm`) is the other opt-in-only backend: it shells out to the `nlm` CLI (notebooklm-mcp-cli; resolved via `GENIUS_NLM_PATH`, keyless — a one-time `nlm login`) and answers from a *configured notebook's* sources (`config.models.notebooklm` / `GENIUS_NOTEBOOKLM_NOTEBOOK` hold the notebook id, since NotebookLM only answers from curated sources); its module-level `nlm_query`/`nlm_list_notebooks`/`nlm_research` coroutines are the shared engine behind the MCP `notebooklm_*` tools.
  - `distributed/hub.py` + `distributed/worker.py` — production `CentralHub` (task queue, routing, heartbeat sweeper, bounded task dict) and `ClientWorker`. These same classes are used in tests via a mock network protocol — do not fork them into test-only variants.
  - `memory/vector_store.py` — `VectorMemory`, with a simple fallback and optional Chroma / sentence-transformers backend.
  - `eval/` — the R5 eval flywheel (grade layer over the traces the pipeline already captures). `metrics.py` (deterministic `CodeExecutionMetric` — `artifacts_present`/`design_wellformed`/`code_syntax_valid`, offline, never executes code — plus LLM-as-judge `LLMMetric` with adaptive rubrics; `BUILTIN_METRICS` registry, `parse_verdict`), `grader.py` (`collect_case` reads a workspace's root artifacts + `*.py` + `logs/raw/` traces; `grade_case`/`grade` score them), `judge.py` (`default_judge` — an in-process provider call via `make_provider`, no new role/port), `compare.py` (before/after regression diff — the gate primitive). The engine behind the MCP `eval` tool.
  - `utils/` — `db.py` (SQLite + WAL, threaded write queue), `security.py` (HMAC-SHA256 checksums + JWT auth), `message_bus.py` (A2A `Artifact` mailbox), `rate_limiter.py`, `git.py`, `jwt.py`, `logger.py`.
- **`.agents/skills/<agent>/`** — per-agent service wrappers: `api.py` (FastAPI app, the `/run` + `/status/{task_id}` background-task pattern, lifespan `init_db`) and `run.py` (standalone CLI runner). `serve.py` loads these dynamically.
- **`mcp_server.py`** — MCP server exposing 18 tools. The **stdio** transport Antigravity consumes is built on the official `mcp` SDK (`_build_sdk_server` + `_run_sdk_stdio`, launched by `python mcp_server.py stdio`) — spec-compliant camelCase `inputSchema`, `protocolVersion` negotiation, and tool failures surfaced as `isError` results; tools are namespaced **`genius_<name>`** on the wire (`WIRE_PREFIX`) to coexist with other MCP servers, while short engine names stay internal (`dispatch_tool("review", ...)`). An optional `GENIUS_MCP_TOOLS` subset selector keeps under Antigravity's ~100-tool cross-server budget. The legacy hand-rolled `handle_request` is retained as the in-process engine the test-suite drives directly, alongside the private FastAPI HTTP tool API (`GET /tools`, `POST /tools/call` — not an MCP transport). The 18 tools: the 6 single agents (research/design/code/unit_test/security_audit/deploy, built in-process via the shared `ag_core.agent_factory`), `orchestrate` (background pipeline job → job_id; `require_approval: true` pauses the sequential pipeline as `awaiting_approval` after the research/design/code stages via `run_pipeline`'s `stage_gate` callback), `orchestrate_approve`/`orchestrate_reject` (resume or fail a paused job; the handlers flip job state synchronously so status polls never see a stale pause), `orchestrate_status` (also reports `elapsed_seconds`, per-stage done/pending `stages` inferred from artifact mtimes vs job start, `artifacts_ready` URIs, and `awaiting_stage` + artifacts while paused), `doctor` (in-process preflight report from `ag_core.diagnostics`), `debate` (in-process researcher-critic ↔ Claude-refiner loop, `[APPROVED]` early exit, max 3 rounds), `review` (codex-role review of supplied code, no file writes) and `code_graph` (read-only CodexGraph-style structure queries over a workspace via `ag_core/scanner/graph_index.py` — op ∈ map/definition/references/importers/imports/skeleton, JSON out, scan+index off the event loop; adding any tool requires updating `tests/test_realrun_mcp.py::EXPECTED_TOOLS`, which pins the exact name set), `eval` (R5 eval flywheel — grade a finished pipeline workspace against metrics via `ag_core/eval/`, in-process, read-only, JSON out, never writes files: op ∈ grade/compare/list_metrics; grade defaults to the deterministic metrics `artifacts_present`/`design_wellformed`/`code_syntax_valid` which run offline, LLM-as-judge metrics `task_success`/`grounding`/`design_quality`/`final_response_quality` are opt-in via `metrics` and call a provider through the shared judge; op=compare diffs two grades and flags regressions — the gate primitive) and three NotebookLM tools — `notebooklm_list` (list notebooks), `notebooklm_query` (grounded, cited answer from an existing notebook's sources) and `notebooklm_research` (MUTATES: web/Drive-discovers sources into a notebook — new unless `notebook` is given — then queries it; synchronous, `fast`≈30s / `deep`≈5min) — all shelling out to the `nlm` CLI via the shared helpers in `ag_core/providers/notebooklm_provider.py` (needs `nlm login` + `GENIUS_NLM_PATH`). Also serves MCP **resources**: a fixed whitelist of root artifacts (`research/design/review/audit/deploy/plan.md` + their `.bak` archives) as `genius://artifacts/<name>` — exact-name match only, never glob (`resources/read` on anything else → `-32002`). stdout must stay pure JSON-RPC; logs go to stderr.

## Ports / roles
Hub `8000`, Researcher `8001`, Claude `8002`, Codex `8003`, Tester `8004`, Security `8005`, DevOps `8006`, Dashboard `8080`. Canonical role id for the Researcher is `researcher` (renamed from the legacy `grok` role id); `grok`/`grok_researcher` are accepted everywhere as compat aliases via `ag_core.provider_factory.canonical_role`. Ports and the other role ids are frozen. The grok BACKEND keeps its name and stays opt-in. Service URLs come from `config.yaml`; a `.agents/service_registry.json` (dynamic port discovery) overrides them at load time.
