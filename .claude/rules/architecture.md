# Architecture

## Request flow
```
Orchestrator  --async httpx + HMAC checksums-->  FastAPI Skill Server  -->  Agent  -->  Provider  -->  local CLI / API
```

- **`orchestrator.py`** ‚Äî the async pipeline runner. `run_pipeline(flow=...)` (sequential; `flow="custom"` is the opt-in user-tailored variant ‚Äî **plan-first** Claude drafts the plan before research, the debate critic is **codex** instead of the researcher, the per-file self-heal has Claude **diagnose** failures before the coder retries, and a **final review** (codex/Fable) ‚Äî which now receives the actual implemented file contents, not just design+audit, so it can't falsely report files "missing" ‚Äî routes issues back to Claude for a fix plan recorded in review.md; an explicitly parsed `{"blocking": true}` verdict then **fails the pipeline** (`final_review_strict()`, default strict; `GENIUS_FINAL_REVIEW_STRICT=0` demotes it to the old advisory record-and-continue, and unparseable reviews stay advisory either way) so a job with known-bad output can never report `completed`. After the per-file fan-out the custom flow additionally runs **whole-project pytest** ONCE from the project root (`_run_project_pytest`; exit 5 = no-tests counts as pass) plus a **design-vs-disk conformance report** (`_design_conformance_report`) ‚Äî both sections are appended to the workspace-root review.md (the single canonical copy) and fed to the final reviewer; a failing suite raises `PipelineError` (`full_suite_gate_enabled()`, default strict, `GENIUS_FULL_SUITE_GATE=0` = report-only) because per-file loops can't see cross-file conflicts ‚Äî audit.md is consolidated and written BEFORE that raise (a real gate failure used to leave no audit.md), and an exit-5 run is annotated "collected NO tests" so a non-Python project can't read as tested. review.md always carries a **"## Verification coverage"** section (`verification_mode`, both flows' review path) breaking files down by how they were actually verified (pytest-executed vs audit-only non-Python/infra) with an explicit WARNING when non-Python files were never executed. Security-audit prompts append `design_scope_note` (the approved design is the CONTRACT: capabilities the design explicitly excludes are accepted risks, NOT blocking ‚Äî a real Next.js job burned all retries on blocking "no auth" findings against a design that forbade auth); `parse_security_verdict` normalizes string `"blocking"` values (`"false"`‚ÜíFalse) so quoted booleans can't invert a gate. The final review records EVERY outcome into review.md (`## Final review (approved)` / `(non-blocking)` / `## Final review` ‚Äî written BEFORE the Claude fix-plan call so a fix-plan crash can't lose the evidence) and raw-captures `final_review`/`final_review_fix_plan`; the MCP custom `review` checkpoint requires the `## Final review` marker in review.md, not just file freshness. Generated test modules dodge basename collisions with DESIGNED test files (`generated_test_path`: `test_<flat>.py` ‚Üí `test_<flat>_gen.py` when the design ships e.g. a root `test_foo.py` ‚Äî both flows; a real job's bare `pytest` failed collection on exactly that). **Deliverable separation:** the project directory (`projects/<slug>/`) holds ONLY designed files ‚Äî pipeline artifacts live at the workspace root only (no project copies), while logs, raw traces, the message-bus DB and generated test modules live under `pipeline_internal_dir()` = `<workspace>/.genius/<slug>/` (fallback `<project>/.genius/` when the project isn't under `projects/`; dot-dir, so pytest collection, the conformance walk and the whole-project gate all skip it). Custom gate order: research ‚Üí design ‚Üí code ‚Üí review ‚Üí devops (the final review runs AFTER the code gate); every custom-only behaviour lives behind `if flow == "custom":` so the default branch is byte-identical ‚Äî `--pipeline custom` on the CLI. The debate critic + final review both target the codex-role service (`codex_url`) by default; when the codex role is repurposed as the coder, `run_pipeline(review_url=‚Ä¶)` or **`GENIUS_REVIEW_ROLE`** (a role name ‚Üí its service URL, e.g. `security`) reroutes them to a codex-gpt5.6-sol service ‚Äî unset ‚áí `codex_url`, byte-identical) and `run_e2e_pipeline()` chain the agents: Researcher (role id `researcher`, agy/Gemini-primary chain) ‚Üí Claude (architect/design) ‚Üí Codex (code) ‚Üí Tester + Security + DevOps. Independent stages run concurrently via `asyncio.gather` (e.g. Tester and Security run in parallel per file in `process_single_file`). Handles tenacity retries, `Retry-After`-aware backoff, background-task polling, and response checksum verification. Per file, a self-healing loop (`--max-retries`, default 3) feeds Tester/Security failures back to Codex; an optional critic‚ÜîClaude design debate (`--max-debate-rounds`, default 2 but **0 under pytest**) exits early on an `[APPROVED]` marker. An unparseable design self-heals in production (`design_selfheal_enabled`, **off under pytest**): up to 2 format re-prompts, then `PipelineError` ‚Äî the legacy single-file branch after it is a PYTEST-ONLY compatibility path, unreachable in production. Files that ARE pytest modules (`is_test_module`) are executed directly instead of getting tests-for-tests + a security audit; pytest infrastructure outside `tests/` (`is_pytest_infra`: root `conftest.py`, package `__init__.py`) and non-Python files get NO generated tests and NO pytest execution ‚Äî their verification is the security audit alone. The e2e variant applies the same three tester-skip guards (non-Python / pytest module / pytest infra). Raw agent responses are captured per stage/attempt under `<workspace>/.genius/<slug>/logs/raw/` (`save_raw_response`). `run_pipeline(stage_gate=...)` awaits an async callback after the research/design/code stages ‚Äî plus review/devops on the custom flow (the MCP approval-gate hook). Codex/Tester output is written through `extract_code(raw, filename=...)` (`ag_core/utils/code_extract.py`): file-type-aware ‚Äî .py starts from the historical python-fence preference and, when that result does NOT `ast.parse`, tries wider fence interpretations (outer-fence unwrap, greedy capture, 4-backtick) accepting the first valid Python (``` sequences INSIDE string literals ‚Äî e.g. a generated `assert "```python" in content` README check ‚Äî otherwise truncate the block mid-string); docs (.md/.rst/.txt) never truncate at nested ``` fences, config types prefer their own language tag; `fence_hint(filename)` supplies the matching per-file prompt instruction. The per-file fan-out is TWO-PHASE (both sequential/custom and e2e): implementation files first, DESIGNED test modules (`is_test_module`) in a second wave once their targets are final ‚Äî racing a mid-rewrite implementation burned a designed test's whole retry budget on a real run; the split lives in `partition_fanout_waves`. **Dependency auto-install** (`GENIUS_AUTO_INSTALL`, opt-in, always off under pytest): the same partition peels root `requirements*.txt` manifests (`is_dependency_manifest`) into a wave 0 ‚Äî implemented first, then `auto_install_requirements` pip-installs them into an isolated `--system-site-packages` venv at `pipeline_internal_dir()/venv`, and every verification subprocess (per-file pytest, e2e flake8/pytest, `_run_project_pytest`) picks its interpreter via `verification_python(project_dir)` (venv when enabled+built, else `sys.executable`). Best-effort: install failures log to the internal `logs/install.log` and never fail the run ‚Äî missing deps surface as ordinary verification failures for the self-heal loop. Off = byte-identical scheduling and interpreters. See configuration.md for the SECURITY caveat.
- **`serve.py`** ‚Äî unified entrypoint. Boots agent API servers by role, the central hub (`--distributed`), or delegates to the orchestrator. Owns the `WorkerRegistry` (idle-worker selection by role) and bounded pending-task tracking.
- **`ag_core/`** ‚Äî the core library:
  - `interfaces/base_agent.py`, `interfaces/base_provider.py` ‚Äî ABCs. `BaseProvider` carries a `TokenBucket` rate limiter + `asyncio.Semaphore(5)`; `BaseAgent` wires in `VectorMemory` and `GitManager`.
  - `directives.py` ‚Äî the `@modifier` layer of the two-layer prompt syntax, separate from the `/cmd` routing/rewrite layers. `parse_directives(text)` strips a leading run of `@deep`/`@critic`/`@variants=N`/`@ideas=N`/format tokens (`@simple|table|steps|tight|natural|redpen`; legacy `/HUMAN`‚Üí`@natural`, `/FLOOD`‚Üí`@ideas`), returning a cleaned prompt whose leading token is still the `/cmd` (so both slash layers see byte-identical input) plus a frozen `PromptDirectives`. It is byte-preserving and returns the ORIGINAL string object when there is nothing to strip (no-op path ‚Äî pinned by `test_slash_commands.py`). `@deep` sets `.effort="high"` (`DEEP_EFFORT`), threaded to the provider as a keyword-only `send_prompt(..., effort=...)` on the call stack ‚Äî never via env ‚Äî so concurrent jobs don't interfere; providers without an effort flag (agy/grok/notebooklm) accept-and-ignore it. `BaseAgent._resolve_user_prompt` parses + gates via the per-agent `ACCEPTED_MODIFIERS` knob (analogous to `SLASH_PREFIXES`): prose agents (researcher/devops) accept the full set; structure-sensitive stages (codex/tester/security/architect) accept only `{deep}` so `@table`/`@variants` can never perturb `extract_code`/`ast.parse`/`parse_security_verdict`/the DesignPlan JSON. Non-effort modifiers render as a trailing `_directive_guidance()` block. Wired on the DIRECT agent-call path (skill server / MCP single-agent / worker) AND pipeline-wide: `orchestrator._resolve_pipeline_setup` parses the leading @modifier run once, returning the cleaned prompt (used only for the slash-command routing DETECTION ‚Äî `first_word`/`ROUTING_TABLE` in orchestrator + serve ‚Äî so `@deep /code ‚Ä¶` still routes to the single /code stage) and the `@deep` effort. The RAW prompt is still what agents receive (they re-parse @modifiers, so prose modifiers keep working), while the effort rides a per-task `_PIPELINE_EFFORT_VAR` contextvar (mirroring `_DISTRIBUTED_MODE_VAR`) that `call_api` reads and folds into the `/run` req_payload + distributed `task_data` (conditionally, before the HMAC, so a None effort is byte-identical); `skill_app.RunRequest.effort` and the worker forward it to `agent.run(effort=‚Ä¶)` ‚Äî so `@deep` reaches codex/claude on structured (synthesized) stages even though those prompts carry no @deep token (agy research is a no-op, logged). Tests: `tests/test_directives.py`, `tests/test_directives_integration.py`, `tests/test_directives_pipeline.py`, per-request effort argv in `test_claude_effort.py`/`test_codex_effort.py`.
  - `agents/` ‚Äî the six agent implementations (researcher, claude_architect, codex_reviewer, tester, security_agent, devops_agent; `grok_researcher.py` is a compat shim re-exporting `ResearcherAgent` as `GrokResearcherAgent`). Their shared request shape (resolve prompt ‚Üí slash-command rewrite ‚Üí scan ‚Üí optional memory context ‚Üí compose ‚Üí send ‚Üí history/memory/log/write) lives in `BaseAgent._run_standard` + class knobs (`SLASH_PREFIXES`, `SYSTEM_PROMPT`, `USES_MEMORY`, `DEFAULT_OUTPUT_FILE`); researcher/architect/security/devops just delegate, codex/tester keep custom `run()`s (verification / self-heal loops) on top of the same helpers. NOTE: `codex_reviewer` must keep a module-level `log_transaction` import ‚Äî tests patch `ag_core.agents.codex_reviewer.log_transaction`.
  - `agent_factory.py` ‚Äî the single agent-construction path (`build_agent(role, ...)`, stateless bundle `output_file="None"/use_memory=False/stateless=True`); `skill_app.ROLE_MAP` and the worker's `ROLE_AGENT_MAP` are derived from its `AGENT_CLASSES`/`LONG_ROLE_ALIASES` tables, and mcp_server passes `agent_cls=globals()[...]` so tests can keep patching `mcp_server.<AgentClass>`.
  - `scanner/` ‚Äî `project_scanner.py` (`ProjectScanner` denylist walk; `ProjectChunker` token counting + greedy `chunk_files`), `repo_graph.py` (graph-aware token-budgeted context, `GENIUS_CONTEXT_TOKEN_BUDGET`), `code_parse.py` (per-language structural parsing: stdlib ast for Python, official tree-sitter grammars for JS/TS/TSX/Go when installed ‚Äî fail-soft to empty info), `graph_index.py` (`RepoIndex`: definition/reference/import indexes + aider-style ranked `repo_map`, the engine behind the MCP `code_graph` tool).
  - `providers/` ‚Äî LLM backends. Each `send_prompt` resolves a local CLI (Windows-aware: `.cmd`/`.bat` are wrapped with `cmd.exe /c`, falls back to `%APPDATA%\npm\`), runs it as a subprocess, and parses JSON/JSONL output. Every role defaults to a `FallbackProvider` chain from `ag_core/provider_factory.py` (Researcher: `agy ‚Üí claude ‚Üí codex`; Architect: `claude ‚Üí agy ‚Üí codex`; others: `codex ‚Üí claude ‚Üí agy`); the grok backend is opt-in only via `GENIUS_PROVIDER_<ROLE>` and auto-runs `grok login` (only when actually invoked) when no API key is present. `notebooklm_provider.py` (backend `notebooklm`) is the other opt-in-only backend: it shells out to the `nlm` CLI (notebooklm-mcp-cli; resolved via `GENIUS_NLM_PATH`, keyless ‚Äî a one-time `nlm login`) and answers from a *configured notebook's* sources (`config.models.notebooklm` / `GENIUS_NOTEBOOKLM_NOTEBOOK` hold the notebook id, since NotebookLM only answers from curated sources); its module-level `nlm_query`/`nlm_list_notebooks`/`nlm_research` coroutines are the shared engine behind the MCP `notebooklm_*` tools.
  - `distributed/hub.py` + `distributed/worker.py` ‚Äî production `CentralHub` (task queue, routing, heartbeat sweeper, bounded task dict) and `ClientWorker`. These same classes are used in tests via a mock network protocol ‚Äî do not fork them into test-only variants.
  - `memory/vector_store.py` ‚Äî `VectorMemory`, with a simple fallback and optional Chroma / sentence-transformers backend.
  - `eval/` ‚Äî the R5 eval flywheel (grade layer over the traces the pipeline already captures). `metrics.py` (deterministic `CodeExecutionMetric` ‚Äî `artifacts_present`/`design_wellformed`/`code_syntax_valid`, offline, never executes code ‚Äî plus LLM-as-judge `LLMMetric` with adaptive rubrics; `BUILTIN_METRICS` registry, `parse_verdict`), `grader.py` (`collect_case` reads a workspace's root artifacts + `*.py` + `logs/raw/` traces; `grade_case`/`grade` score them), `judge.py` (`default_judge` ‚Äî an in-process provider call via `make_provider`, no new role/port), `compare.py` (before/after regression diff ‚Äî the gate primitive). The engine behind the MCP `eval` tool.
  - `utils/` ‚Äî `db.py` (SQLite + WAL, threaded write queue), `security.py` (HMAC-SHA256 checksums + JWT auth), `message_bus.py` (A2A `Artifact` mailbox), `code_extract.py` (file-type-aware `extract_code` + `fence_hint`, tests in `tests/test_code_extract.py`), `rate_limiter.py`, `git.py`, `jwt.py`, `logger.py`.
- **`.agents/skills/<agent>/`** ‚Äî per-agent service wrappers: `api.py` (FastAPI app, the `/run` + `/status/{task_id}` background-task pattern, lifespan `init_db`) and `run.py` (standalone CLI runner). `serve.py` loads these dynamically.
- **`mcp_server.py`** ‚Äî MCP server exposing 18 tools. The **stdio** transport Antigravity consumes is built on the official `mcp` SDK (`_build_sdk_server` + `_run_sdk_stdio`, launched by `python mcp_server.py stdio`) ‚Äî spec-compliant camelCase `inputSchema`, `protocolVersion` negotiation, and tool failures surfaced as `isError` results; tools are namespaced **`genius_<name>`** on the wire (`WIRE_PREFIX`) to coexist with other MCP servers, while short engine names stay internal (`dispatch_tool("review", ...)`). An optional `GENIUS_MCP_TOOLS` subset selector keeps under Antigravity's ~100-tool cross-server budget. The legacy hand-rolled `handle_request` is retained as the in-process engine the test-suite drives directly, alongside the private FastAPI HTTP tool API (`GET /tools`, `POST /tools/call` ‚Äî not an MCP transport). The 18 tools: the 6 single agents (research/design/code/unit_test/security_audit/deploy, built in-process via the shared `ag_core.agent_factory`), `orchestrate` (background pipeline job ‚Üí job_id; `require_approval: true` pauses the pipeline as `awaiting_approval` after the research/design/code stages ‚Äî plus review/devops on the custom flow ‚Äî via `run_pipeline`'s `stage_gate` callback), `orchestrate_approve`/`orchestrate_reject` (resume or fail a paused job; the handlers flip job state synchronously so status polls never see a stale pause), `orchestrate_status` (also reports `elapsed_seconds`, `workspace`, `current_stage` ‚Äî the first not-yet-done checkpoint while running ‚Äî per-stage done/pending `stages` inferred from artifact mtimes vs job start, `artifacts_ready` URIs, and `awaiting_stage` + artifacts while paused; every job journals a `job.json` manifest into its workspace on each state transition, and an id no longer in memory is recovered from `<jobs dir>/<job_id>/job.json` ‚Äî a manifest still saying running/awaiting_approval is reported as `interrupted` with `recovered_from_journal: true`. Each orchestrate call also prunes old job dirs under the jobs root: `GENIUS_JOBS_MAX_JOBS` newest kept (default 50, 0 off) + opt-in `GENIUS_JOBS_RETENTION_DAYS`; only 32-hex dirs are candidates and unfinished in-memory jobs are never touched), `doctor` (in-process preflight report from `ag_core.diagnostics`), `debate` (in-process researcher-critic ‚Üî Claude-refiner loop, `[APPROVED]` early exit, max 3 rounds), `review` (codex-role review of supplied code, no file writes) and `code_graph` (read-only CodexGraph-style structure queries over a workspace via `ag_core/scanner/graph_index.py` ‚Äî op ‚àà map/definition/references/importers/imports/skeleton, JSON out, scan+index off the event loop; adding any tool requires updating `tests/test_realrun_mcp.py::EXPECTED_TOOLS`, which pins the exact name set), `eval` (R5 eval flywheel ‚Äî grade a finished pipeline workspace against metrics via `ag_core/eval/`, in-process, read-only, JSON out, never writes files: op ‚àà grade/compare/list_metrics; grade defaults to the deterministic metrics `artifacts_present`/`design_wellformed`/`code_syntax_valid` which run offline, LLM-as-judge metrics `task_success`/`grounding`/`design_quality`/`final_response_quality` are opt-in via `metrics` and call a provider through the shared judge; op=compare diffs two grades and flags regressions ‚Äî the gate primitive) and three NotebookLM tools ‚Äî `notebooklm_list` (list notebooks), `notebooklm_query` (grounded, cited answer from an existing notebook's sources) and `notebooklm_research` (MUTATES: web/Drive-discovers sources into a notebook ‚Äî new unless `notebook` is given ‚Äî then queries it; synchronous, `fast`‚âà30s / `deep`‚âà5min) ‚Äî all shelling out to the `nlm` CLI via the shared helpers in `ag_core/providers/notebooklm_provider.py` (needs `nlm login` + `GENIUS_NLM_PATH`). While the stdio transport is live, a per-job watcher additionally PUSHES stage completions and status changes as spec-standard `notifications/message` log notifications (logger `genius.orchestrate`; the `logging` capability is declared and `logging/setLevel` raises the floor) ‚Äî polling is unchanged, the push is additive. The pipeline itself honors opt-in adaptive effort (`GENIUS_ADAPTIVE_EFFORT`): prompts under ~600 chars run at effort `high` instead of the per-role env defaults unless `@deep` is given. Also serves MCP **resources**: a fixed whitelist of pipeline artifacts (`research/design/review/audit/deploy/plan.md` + their `.bak` archives), never a glob, in two URI forms ‚Äî **job-scoped** `genius://artifacts/<job_id>/<name>` (what `orchestrate_status.artifacts_ready` advertises for real 32-hex job ids; resolved ONLY inside that job's workspace via `ag_core/mcp_resources.py`'s registry + a `job.json` journal resolver, so a stale root artifact or a concurrent job can never shadow the read) and **legacy bare** `genius://artifacts/<name>` (CWD first, then the last job observed to produce that name). Exact whitelist-name/32-hex-id match only (`resources/read` on anything else ‚Üí `-32002`). stdout must stay pure JSON-RPC; logs go to stderr.

## Ports / roles
Hub `8000`, Researcher `8001`, Claude `8002`, Codex `8003`, Tester `8004`, Security `8005`, DevOps `8006`, Dashboard `8080`. Canonical role id for the Researcher is `researcher` (renamed from the legacy `grok` role id); `grok`/`grok_researcher` are accepted everywhere as compat aliases via `ag_core.provider_factory.canonical_role`. Ports and the other role ids are frozen. The grok BACKEND keeps its name and stays opt-in. Service URLs come from `config.yaml`; a `.agents/service_registry.json` (dynamic port discovery) overrides them at load time.
